## Data & Formats — ข้อมูลและรูปแบบไฟล์

### Open Data
**ภาษาไทย:** ข้อมูลเปิด
ข้อมูลที่เผยแพร่ให้สาธารณะสามารถเข้าถึง ใช้งาน และแจกจ่ายได้อย่างเสรี โดยไม่มีข้อจำกัดด้านลิขสิทธิ์หรือสิทธิบัตร
ในโปรเจกต์นี้ ข้อมูลสถานีบริการน้ำมันจาก Thai Pump Radar ถือเป็น Open Data ที่เราใช้เป็นวัตถุดิบตั้งต้น

### AI-Ready Data
**ภาษาไทย:** ข้อมูลพร้อมใช้งาน AI
ข้อมูลที่ผ่านการทำความสะอาด จัดโครงสร้าง เพิ่ม Metadata และ Embedding จนพร้อมสำหรับการนำไปใช้กับโมเดล AI ได้ทันที
Level 6 หมายถึงระดับสูงสุดที่ข้อมูลมี Semantic Embedding, Data Card และสามารถค้นหาเชิงความหมายได้

### Data Card
**ภาษาไทย:** บัตรข้อมูล / การ์ดข้อมูล
เอกสารมาตรฐานที่อธิบายรายละเอียดของชุดข้อมูล เช่น ที่มา วิธีการเก็บ ข้อจำกัด และวิธีใช้งาน
ทำหน้าที่เหมือน "ฉลากสินค้า" ของข้อมูล ช่วยให้ผู้ใช้เข้าใจบริบทก่อนนำข้อมูลไปใช้

### Data Lake
**ภาษาไทย:** ทะเลสาบข้อมูล
ที่เก็บข้อมูลขนาดใหญ่ที่รับข้อมูลดิบทุกรูปแบบ (structured, semi-structured, unstructured) เข้ามาเก็บก่อนโดยยังไม่ต้องจัดโครงสร้าง
เหมาะกับการเก็บข้อมูลดิบจาก API ของ Pump Radar ก่อนนำไปแปลงในขั้นตอนถัดไป

### Data Warehouse
**ภาษาไทย:** คลังข้อมูล
ที่เก็บข้อมูลที่ผ่านการจัดโครงสร้างและทำความสะอาดแล้ว ออกแบบมาเพื่อการวิเคราะห์และสืบค้นอย่างมีประสิทธิภาพ
ต่างจาก Data Lake ตรงที่ข้อมูลใน Warehouse มี Schema ชัดเจนและพร้อมใช้งานทันที

### JSON (JavaScript Object Notation)
**ภาษาไทย:** เจสัน
รูปแบบข้อมูลแบบ key-value ที่อ่านง่ายทั้งคนและเครื่อง ใช้กันแพร่หลายในการรับส่งข้อมูลผ่าน API
ข้อมูลจาก Pump Radar API จะได้มาในรูปแบบ JSON เป็นหลัก

### JSONL (JSON Lines)
**ภาษาไทย:** เจสันไลน์
รูปแบบที่แต่ละบรรทัดเป็น JSON object หนึ่งชิ้น เหมาะกับการประมวลผลข้อมูลขนาดใหญ่แบบทีละแถว (streaming)
ในโปรเจกต์นี้ใช้ JSONL สำหรับจัดเก็บข้อมูลสถานีน้ำมันที่แปลงแล้ว เพราะอ่านทีละบรรทัดได้โดยไม่ต้องโหลดทั้งไฟล์

### Parquet
**ภาษาไทย:** พาร์เก้
รูปแบบไฟล์แบบ columnar ที่บีบอัดข้อมูลได้ดีและอ่านเร็ว เหมาะกับงาน analytics และ big data
ใช้เป็นรูปแบบจัดเก็บข้อมูลพร้อมใช้งาน AI เพราะประหยัดพื้นที่และรองรับ schema ที่ซับซ้อนได้

### CSV (Comma-Separated Values)
**ภาษาไทย:** ซีเอสวี
รูปแบบข้อมูลตารางอย่างง่ายที่คั่นค่าด้วยจุลภาค เปิดได้ด้วยโปรแกรม spreadsheet ทั่วไป
เหมาะสำหรับแชร์ข้อมูลให้คนทั่วไป แต่มีข้อจำกัดเรื่อง data type และโครงสร้างซ้อน

### RDF (Resource Description Framework)
**ภาษาไทย:** อาร์ดีเอฟ
มาตรฐานสำหรับอธิบายข้อมูลในรูปแบบ subject-predicate-object (triple) ใช้ในงาน Linked Data และ Semantic Web
ช่วยให้ข้อมูลสถานีน้ำมันเชื่อมโยงกับชุดข้อมูลอื่นได้อย่างมีความหมาย เช่น เชื่อมกับข้อมูลภูมิศาสตร์

### Schema
**ภาษาไทย:** สคีมา / โครงสร้างข้อมูล
คำอธิบายโครงสร้างของข้อมูล กำหนดว่ามี field อะไรบ้าง แต่ละ field เป็นชนิดข้อมูลอะไร และมีข้อจำกัดอย่างไร
ในโปรเจกต์นี้ Schema กำหนดโครงสร้างข้อมูลสถานีน้ำมัน เช่น ชื่อสถานี ที่อยู่ พิกัด ราคาน้ำมัน

### Metadata
**ภาษาไทย:** เมตาดาต้า / ข้อมูลอธิบายข้อมูล
ข้อมูลที่อธิบายคุณลักษณะของข้อมูลอื่น เช่น วันที่สร้าง แหล่งที่มา รูปแบบ และเงื่อนไขการใช้งาน
เป็นส่วนสำคัญของ Data Card ที่ช่วยให้ AI เข้าใจบริบทของชุดข้อมูลก่อนนำไปใช้

### Field
**ภาษาไทย:** ฟิลด์ / คอลัมน์
หน่วยย่อยที่สุดของโครงสร้างข้อมูลที่เก็บค่าข้อมูลประเภทเดียวกัน เช่น ชื่อสถานี ราคาน้ำมัน หรือพิกัด GPS
เทียบได้กับคอลัมน์ในตาราง Excel

### Record
**ภาษาไทย:** เรคคอร์ด / แถวข้อมูล
ชุดข้อมูลหนึ่งรายการที่ประกอบด้วย field ต่าง ๆ ครบถ้วน เช่น ข้อมูลสถานีน้ำมัน 1 สถานี = 1 record
เทียบได้กับแถวหนึ่งแถวในตาราง Excel

## Data Engineering — วิศวกรรมข้อมูล

### Data Ingestion
**ภาษาไทย:** การนำเข้าข้อมูล
กระบวนการดึงข้อมูลจากแหล่งต้นทางเข้ามาในระบบ ไม่ว่าจะจาก API, ไฟล์ หรือฐานข้อมูล
ในโปรเจกต์นี้คือขั้นตอนการดึงข้อมูลสถานีน้ำมันจาก Pump Radar API เข้ามาในระบบ pipeline

### ETL (Extract-Transform-Load)
**ภาษาไทย:** สกัด-แปลง-โหลด
กระบวนการ 3 ขั้นตอน: สกัดข้อมูลจากต้นทาง (Extract) แปลงข้อมูลให้อยู่ในรูปแบบที่ต้องการ (Transform) แล้วโหลดเข้าที่เก็บปลายทาง (Load)
ในโปรเจกต์นี้ ETL แปลงข้อมูลดิบจาก JSON เป็น AI-Ready Data ในรูปแบบ Parquet พร้อม Embedding

### ELT (Extract-Load-Transform)
**ภาษาไทย:** สกัด-โหลด-แปลง
รูปแบบที่โหลดข้อมูลดิบเข้าที่เก็บก่อน แล้วค่อยแปลงทีหลัง เหมาะกับระบบที่มีพลังประมวลผลสูงที่ปลายทาง
ต่างจาก ETL ตรงที่ข้อมูลดิบจะถูกเก็บไว้ใน Data Lake ก่อน แล้วจึงแปลงตามความต้องการในภายหลัง

### Data Pipeline
**ภาษาไทย:** ท่อส่งข้อมูล / ไปป์ไลน์ข้อมูล
ชุดขั้นตอนอัตโนมัติที่เชื่อมต่อกันเป็นลำดับ ตั้งแต่การนำเข้า แปลง ตรวจสอบ จนถึงส่งออกข้อมูล
โปรเจกต์นี้มี pipeline ที่ทำงานตั้งแต่ดึงข้อมูลจาก API จนถึงสร้าง Embedding และ Data Card

### Batch Processing
**ภาษาไทย:** การประมวลผลเป็นชุด
การประมวลผลข้อมูลจำนวนมากในคราวเดียว เช่น ประมวลผลข้อมูลสถานีน้ำมันทั้งหมดทุกคืน
เหมาะกับข้อมูลที่ไม่จำเป็นต้องอัปเดตแบบ real-time เช่น ข้อมูลราคาน้ำมันรายวัน

### Streaming
**ภาษาไทย:** การประมวลผลแบบสตรีม
การประมวลผลข้อมูลทันทีที่ข้อมูลเข้ามา โดยไม่ต้องรอสะสมเป็นชุดก่อน
เหมาะกับข้อมูลที่ต้องการความ real-time เช่น การอัปเดตราคาน้ำมันที่เปลี่ยนแปลงบ่อย

### Event-driven
**ภาษาไทย:** ขับเคลื่อนด้วยเหตุการณ์
สถาปัตยกรรมที่ทำงานเมื่อเกิดเหตุการณ์บางอย่าง แทนที่จะทำงานตามเวลาที่กำหนด
เช่น pipeline เริ่มทำงานเมื่อมีข้อมูลใหม่จาก API แทนที่จะรันทุก ๆ ชั่วโมง

### Data Cleaning
**ภาษาไทย:** การทำความสะอาดข้อมูล
กระบวนการแก้ไขหรือลบข้อมูลที่ผิดพลาด ไม่สมบูรณ์ หรือซ้ำซ้อนออกจากชุดข้อมูล
เช่น การแก้ชื่อสถานีน้ำมันที่สะกดผิด ลบรายการที่ไม่มีพิกัด GPS หรือแปลงรูปแบบที่อยู่ให้สม่ำเสมอ

### Data Validation
**ภาษาไทย:** การตรวจสอบความถูกต้องของข้อมูล
การตรวจสอบว่าข้อมูลตรงตามเงื่อนไขที่กำหนด เช่น ประเภทข้อมูลถูกต้อง ค่าอยู่ในช่วงที่เป็นไปได้ และไม่ขาด field สำคัญ
เช่น ตรวจสอบว่าพิกัด GPS อยู่ในพื้นที่ประเทศไทย หรือราคาน้ำมันเป็นจำนวนบวก

### Normalization
**ภาษาไทย:** การทำให้เป็นมาตรฐาน / นอร์มัลไลเซชัน
การปรับข้อมูลให้อยู่ในรูปแบบและมาตรฐานเดียวกัน เช่น ปรับหน่วยวัด รูปแบบวันที่ หรือการเข้ารหัสตัวอักษร
เช่น ทำให้ชื่อจังหวัดทั้งหมดใช้รูปแบบเดียวกัน ("กรุงเทพฯ" กับ "กรุงเทพมหานคร" ให้เป็นค่าเดียว)

### De-duplication
**ภาษาไทย:** การกำจัดข้อมูลซ้ำ
การค้นหาและลบข้อมูลที่ซ้ำกันออก เพื่อให้ชุดข้อมูลมีความถูกต้องและไม่บวมเกินจริง
เช่น สถานีน้ำมันเดียวกันที่ถูกบันทึกซ้ำหลายครั้งด้วยชื่อต่างกันเล็กน้อย

### Confidence Score
**ภาษาไทย:** คะแนนความเชื่อมั่น
ค่าตัวเลข (มักเป็น 0-1) ที่บ่งบอกระดับความน่าเชื่อถือของข้อมูลหรือผลลัพธ์
ในโปรเจกต์นี้ใช้ Confidence Score เพื่อระบุว่าข้อมูลสถานีน้ำมันแต่ละรายการมีความสมบูรณ์และน่าเชื่อถือมากน้อยเพียงใด

## AI & Machine Learning — ปัญญาประดิษฐ์และการเรียนรู้ของเครื่อง

### RAG (Retrieval-Augmented Generation)
**ภาษาไทย:** การสร้างคำตอบโดยเสริมด้วยการค้นคืน
เทคนิคที่ดึงข้อมูลที่เกี่ยวข้องจากฐานความรู้มาประกอบก่อนให้ LLM สร้างคำตอบ ช่วยให้คำตอบถูกต้องและมีแหล่งอ้างอิง
ในโปรเจกต์นี้ RAG ช่วยให้ AI ตอบคำถามเกี่ยวกับสถานีน้ำมันได้ถูกต้อง โดยดึงข้อมูลจริงมาประกอบ

### Embedding
**ภาษาไทย:** เอ็มเบดดิง / การแปลงเป็นเวกเตอร์
การแปลงข้อมูล (ข้อความ รูปภาพ ฯลฯ) ให้เป็นเวกเตอร์ตัวเลขที่เก็บรักษาความหมายเชิงความหมายไว้
ข้อมูลสถานีน้ำมันแต่ละแห่งจะถูกแปลงเป็น Embedding เพื่อให้ค้นหาเชิงความหมายได้ เช่น "ปั๊มใกล้สนามบิน"

### Vector
**ภาษาไทย:** เวกเตอร์
ชุดตัวเลขที่เรียงกันเป็นมิติ ใช้แทนข้อมูลในรูปแบบที่คอมพิวเตอร์ประมวลผลได้
เป็นผลลัพธ์จากกระบวนการ Embedding โดยข้อมูลที่มีความหมายใกล้เคียงกันจะมี Vector ที่อยู่ใกล้กัน

### Vector Database
**ภาษาไทย:** ฐานข้อมูลเวกเตอร์
ฐานข้อมูลที่ออกแบบมาเพื่อจัดเก็บและค้นหา Vector ได้อย่างรวดเร็ว รองรับการค้นหาเชิงความหมาย
ใช้เก็บ Embedding ของข้อมูลสถานีน้ำมัน เพื่อให้ MCP Server ค้นหาสถานีที่เกี่ยวข้องกับคำถามของผู้ใช้ได้

### Semantic Search
**ภาษาไทย:** การค้นหาเชิงความหมาย
การค้นหาข้อมูลโดยเข้าใจ "ความหมาย" ของคำค้นหา ไม่ใช่แค่จับคู่คำตรงตัว
เช่น ค้นคำว่า "ปั๊มน้ำมันใกล้ทะเล" สามารถค้นพบสถานีที่อยู่ในจังหวัดชายทะเลได้ แม้ข้อมูลไม่ได้เขียนคำว่า "ทะเล"

### Keyword Search
**ภาษาไทย:** การค้นหาด้วยคำสำคัญ
การค้นหาข้อมูลโดยจับคู่คำตรงตัว (exact match) หรือคำที่ตรงกับ index
เร็วและแม่นยำสำหรับการค้นหาที่ชัดเจน เช่น ค้นชื่อสถานีน้ำมันเฉพาะแห่ง แต่ไม่เข้าใจความหมายเชิงบริบท

### Cosine Similarity
**ภาษาไทย:** ความคล้ายคลึงโคไซน์
วิธีวัดความใกล้เคียงระหว่าง Vector สองตัว โดยดูมุมระหว่างเวกเตอร์ ค่า 1 หมายถึงเหมือนกัน ค่า 0 หมายถึงไม่เกี่ยวข้อง
ใช้ในการจัดอันดับผลลัพธ์การค้นหาเชิงความหมาย เช่น หาสถานีน้ำมันที่ตรงกับคำถามผู้ใช้มากที่สุด

### LLM (Large Language Model)
**ภาษาไทย:** โมเดลภาษาขนาดใหญ่
โมเดล AI ที่ฝึกมาจากข้อความจำนวนมหาศาล สามารถเข้าใจและสร้างข้อความภาษาธรรมชาติได้ เช่น GPT, Claude, Gemini
ในโปรเจกต์นี้ LLM ทำงานร่วมกับ MCP Server เพื่อตอบคำถามเกี่ยวกับข้อมูลสถานีน้ำมัน

### Fine-tuning
**ภาษาไทย:** การปรับจูนโมเดล
การนำ LLM ที่ฝึกมาแล้วมาฝึกเพิ่มเติมด้วยข้อมูลเฉพาะทาง เพื่อให้ทำงานได้ดีขึ้นในบริบทที่ต้องการ
เป็นทางเลือกนอกเหนือจาก RAG สำหรับการทำให้ AI เข้าใจข้อมูลเฉพาะด้าน แต่ต้องใช้ทรัพยากรมากกว่า

### Prompt Engineering
**ภาษาไทย:** วิศวกรรมพรอมต์
ศาสตร์ของการออกแบบคำสั่ง (prompt) ที่ให้กับ LLM เพื่อให้ได้ผลลัพธ์ที่ดีที่สุด
ในโปรเจกต์นี้ Prompt Engineering ช่วยออกแบบวิธีที่ MCP Server ส่งบริบทข้อมูลสถานีน้ำมันให้ LLM ประมวลผล

### Knowledge Base
**ภาษาไทย:** ฐานความรู้
แหล่งรวมข้อมูลที่จัดระเบียบแล้ว ใช้เป็นแหล่งอ้างอิงสำหรับ AI ในการตอบคำถาม
ข้อมูล AI-Ready ของสถานีน้ำมันที่ผ่าน pipeline แล้ว จะถูกจัดเก็บเป็น Knowledge Base สำหรับ RAG

## MCP & Protocols — โปรโตคอล MCP

### MCP (Model Context Protocol)
**ภาษาไทย:** โปรโตคอลบริบทโมเดล
โปรโตคอลมาตรฐานเปิดที่กำหนดวิธีการสื่อสารระหว่าง LLM กับแหล่งข้อมูลภายนอก ช่วยให้ AI เข้าถึงข้อมูลได้อย่างปลอดภัยและเป็นระบบ
เป็นหัวใจของโปรเจกต์นี้ ทำหน้าที่เป็นสะพานเชื่อมระหว่าง AI กับข้อมูลสถานีน้ำมัน

### MCP Tool
**ภาษาไทย:** เครื่องมือ MCP
ฟังก์ชันที่ MCP Server เปิดให้ LLM เรียกใช้ได้ เช่น ค้นหาสถานีน้ำมัน ดึงราคาน้ำมัน หรือดูข้อมูลจังหวัด
แต่ละ Tool มี schema กำหนด input/output ชัดเจน ให้ LLM รู้ว่าเครื่องมือนี้ทำอะไรได้บ้าง

### MCP Server
**ภาษาไทย:** เซิร์ฟเวอร์ MCP
โปรแกรมที่ให้บริการข้อมูลและเครื่องมือผ่าน MCP Protocol รับคำขอจาก LLM แล้วส่งผลลัพธ์กลับ
โปรเจกต์นี้สร้าง MCP Server ที่ให้บริการข้อมูลสถานีน้ำมันพร้อมใช้งาน AI

### Stdio (Standard Input/Output)
**ภาษาไทย:** อินพุต/เอาต์พุตมาตรฐาน
วิธีการสื่อสารผ่าน stdin/stdout ของโปรเซส ใช้สำหรับ MCP Server ที่รันเป็น local process
เป็น transport แบบง่ายที่สุด เหมาะกับการพัฒนาและทดสอบ MCP Server บนเครื่องของนักพัฒนาเอง

### SSE (Server-Sent Events)
**ภาษาไทย:** การส่งเหตุการณ์จากเซิร์ฟเวอร์
เทคโนโลยีที่เซิร์ฟเวอร์ส่งข้อมูลไปยัง client แบบทิศทางเดียวผ่าน HTTP เหมาะกับการส่ง stream ของข้อมูล
ใช้เป็น transport สำหรับ MCP Server ที่ต้องการส่งผลลัพธ์แบบ real-time

### Streamable HTTP
**ภาษาไทย:** HTTP แบบสตรีมได้
transport รูปแบบใหม่ของ MCP ที่รวมข้อดีของ HTTP และ SSE เข้าด้วยกัน รองรับทั้งการส่งคำขอแบบปกติและการ stream ผลลัพธ์
เป็นทิศทางที่ MCP กำลังพัฒนาไป เพื่อรองรับการใช้งานจริงบน production ได้ดีขึ้น

### Transport
**ภาษาไทย:** ช่องทางการสื่อสาร / ทรานสปอร์ต
วิธีการรับส่งข้อมูลระหว่าง MCP Client กับ MCP Server เช่น Stdio, SSE หรือ Streamable HTTP
การเลือก transport ขึ้นอยู่กับสภาพแวดล้อม เช่น local development ใช้ Stdio ส่วน production ใช้ HTTP

### JSON-RPC
**ภาษาไทย:** เจสัน-อาร์พีซี
โปรโตคอลสำหรับเรียกใช้ฟังก์ชันระยะไกล (Remote Procedure Call) โดยใช้ JSON เป็นรูปแบบข้อความ
MCP ใช้ JSON-RPC เป็นรูปแบบข้อความพื้นฐานในการสื่อสารระหว่าง client กับ server

### FastMCP
**ภาษาไทย:** ฟาสต์เอ็มซีพี
เฟรมเวิร์ก Python ที่ช่วยสร้าง MCP Server ได้ง่ายและรวดเร็ว ลด boilerplate code ลงมาก
ในโปรเจกต์นี้ใช้ FastMCP เป็นเครื่องมือหลักในการสร้าง MCP Server สำหรับให้บริการข้อมูลสถานีน้ำมัน

## Infrastructure — โครงสร้างพื้นฐาน

### Docker
**ภาษาไทย:** ด็อกเกอร์
เครื่องมือสร้างและจัดการ container ที่ห่อหุ้มแอปพลิเคชันพร้อม dependencies ทั้งหมด ทำให้รันได้เหมือนกันทุกเครื่อง
ใช้ในโปรเจกต์นี้เพื่อให้ MCP Server และ pipeline ทำงานได้โดยไม่ต้องกังวลเรื่องสภาพแวดล้อม

### Docker Compose
**ภาษาไทย:** ด็อกเกอร์ คอมโพส
เครื่องมือสำหรับจัดการหลาย container พร้อมกันด้วยไฟล์ YAML ไฟล์เดียว
ในโปรเจกต์นี้ใช้ docker-compose.yml กำหนดการทำงานของ MCP Server และบริการที่เกี่ยวข้องทั้งหมด

### Container
**ภาษาไทย:** คอนเทนเนอร์
สภาพแวดล้อมที่แยกออกมา (isolated) สำหรับรันแอปพลิเคชัน เบากว่า virtual machine เพราะใช้ kernel ร่วมกับ host
แต่ละ container มีระบบไฟล์ เครือข่าย และ process ของตัวเอง ทำให้ไม่กระทบกับ container อื่น

### Image
**ภาษาไทย:** อิมเมจ
แม่แบบแบบอ่านอย่างเดียว (read-only template) ที่ใช้สร้าง container ประกอบด้วย OS, dependencies และโค้ดแอปพลิเคชัน
สร้างจาก Dockerfile ในโปรเจกต์นี้ และเป็นต้นแบบที่ใช้สร้าง container ทุกครั้ง

### Volume
**ภาษาไทย:** วอลุ่ม
พื้นที่จัดเก็บข้อมูลถาวรที่แยกจาก container ทำให้ข้อมูลไม่หายเมื่อ container ถูกลบ
ใช้เก็บข้อมูลสถานีน้ำมัน ไฟล์ Parquet และ Embedding ที่ต้องคงอยู่แม้ container ถูกสร้างใหม่

### GitHub Codespaces
**ภาษาไทย:** กิตฮับ โค้ดสเปซ
สภาพแวดล้อมการพัฒนาบนคลาวด์ที่ GitHub ให้บริการ เปิดใช้งานได้จากเบราว์เซอร์โดยไม่ต้องติดตั้งอะไรในเครื่อง
ช่วยให้นักพัฒนาเริ่มทำงานกับโปรเจกต์ได้ทันทีโดยไม่ต้องตั้งค่าสภาพแวดล้อมเอง

### DevContainer
**ภาษาไทย:** เดฟคอนเทนเนอร์
ไฟล์กำหนดค่าที่ระบุสภาพแวดล้อมการพัฒนาในรูปแบบ container ใช้กับ VS Code และ GitHub Codespaces
กำหนด extensions, dependencies และการตั้งค่าทั้งหมดที่จำเป็นสำหรับโปรเจกต์ไว้ในไฟล์เดียว

### API (Application Programming Interface)
**ภาษาไทย:** เอพีไอ / ส่วนต่อประสานโปรแกรม
ช่องทางที่โปรแกรมใช้สื่อสารกัน กำหนดรูปแบบการเรียกใช้และรูปแบบข้อมูลที่รับส่ง
Pump Radar เปิด API ให้ดึงข้อมูลสถานีน้ำมัน และ MCP Server เองก็เปิด API ให้ LLM เรียกใช้

### REST API
**ภาษาไทย:** เรสต์ เอพีไอ
รูปแบบการออกแบบ API ที่ใช้ HTTP methods (GET, POST, PUT, DELETE) ในการจัดการทรัพยากร
เป็นรูปแบบ API ที่ใช้กันแพร่หลายที่สุด และเป็นรูปแบบที่ Pump Radar ใช้ให้บริการข้อมูล

### Endpoint
**ภาษาไทย:** เอนด์พอยต์ / จุดเชื่อมต่อ
URL เฉพาะที่ API เปิดให้เรียกใช้ แต่ละ endpoint ทำหน้าที่เฉพาะอย่าง เช่น ดึงรายชื่อสถานี หรือค้นหาตามจังหวัด
เช่น `/api/stations` เป็น endpoint สำหรับดึงข้อมูลสถานีน้ำมันทั้งหมด

## Data Governance — ธรรมาภิบาลข้อมูล

### PDPA (Personal Data Protection Act)
**ภาษาไทย:** พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล
กฎหมายไทยที่กำกับดูแลการเก็บรวบรวม ใช้ และเปิดเผยข้อมูลส่วนบุคคล มีผลบังคับใช้ตั้งแต่ พ.ศ. 2565
ในโปรเจกต์นี้ต้องมั่นใจว่าข้อมูลสถานีน้ำมันไม่มีข้อมูลส่วนบุคคลปนอยู่ หรือถ้ามีต้องจัดการตาม PDPA

### PII (Personally Identifiable Information)
**ภาษาไทย:** ข้อมูลที่สามารถระบุตัวบุคคลได้
ข้อมูลที่ใช้ระบุตัวตนบุคคลได้ เช่น ชื่อ-นามสกุล เลขบัตรประชาชน เบอร์โทรศัพท์ ที่อยู่อีเมล
ต้องตรวจสอบและลบ PII ออกจากข้อมูลก่อนนำไปเผยแพร่เป็น Open Data หรือ AI-Ready Data

### Data Steward
**ภาษาไทย:** ผู้ดูแลข้อมูล
ผู้รับผิดชอบในการดูแลคุณภาพ ความถูกต้อง และการใช้งานข้อมูลให้เป็นไปตามนโยบาย
ทำหน้าที่เป็นคนกลางระหว่าง Data Owner กับผู้ใช้ข้อมูล คอยตรวจสอบว่าข้อมูลถูกใช้อย่างเหมาะสม

### Data Owner
**ภาษาไทย:** เจ้าของข้อมูล
ผู้มีอำนาจตัดสินใจเกี่ยวกับข้อมูลชุดนั้น กำหนดว่าใครเข้าถึงได้ ใช้ทำอะไรได้ และมีเงื่อนไขอะไร
ในกรณีข้อมูล Pump Radar เจ้าของข้อมูลคือหน่วยงานที่จัดเก็บและเผยแพร่ข้อมูลสถานีน้ำมัน

### Lineage
**ภาษาไทย:** สายข้อมูล / เส้นทางข้อมูล
การติดตามว่าข้อมูลมาจากไหน ผ่านกระบวนการอะไรบ้าง และถูกแปลงอย่างไรจนมาถึงสถานะปัจจุบัน
ช่วยให้ตรวจสอบย้อนกลับได้ว่าข้อมูล AI-Ready ของสถานีน้ำมันแต่ละรายการมาจากข้อมูลดิบชุดไหน

### Provenance
**ภาษาไทย:** ที่มาของข้อมูล
ข้อมูลที่บอกแหล่งกำเนิด ประวัติการเปลี่ยนแปลง และความเป็นเจ้าของของข้อมูล
คล้ายกับ Lineage แต่เน้นที่ "ใคร" ทำ "อะไร" กับข้อมูล "เมื่อไหร่" มากกว่าเส้นทางทางเทคนิค

### Bias
**ภาษาไทย:** อคติในข้อมูล
ความเอนเอียงที่เกิดจากข้อมูลที่ไม่ครบถ้วนหรือไม่เป็นตัวแทนของความเป็นจริง ส่งผลให้ AI ตัดสินใจผิดพลาด
เช่น ถ้าข้อมูลสถานีน้ำมันมีแต่ในกรุงเทพฯ AI อาจตอบคำถามเกี่ยวกับสถานีในต่างจังหวัดไม่ได้

### Digital Trust
**ภาษาไทย:** ความไว้วางใจทางดิจิทัล
ความเชื่อมั่นที่ผู้ใช้มีต่อระบบดิจิทัลว่าข้อมูลถูกต้อง ปลอดภัย โปร่งใส และเป็นธรรม
การมี Data Card, Lineage และ Confidence Score ช่วยสร้าง Digital Trust ให้กับชุดข้อมูล AI-Ready ของโปรเจกต์นี้
